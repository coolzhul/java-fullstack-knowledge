## ⭐ 高并发场景下有哪些限流方案？令牌桶和漏桶算法的区别？

**简要回答：** 常见限流方案有：固定窗口/滑动窗口计数器、漏桶算法、令牌桶算法。令牌桶允许突发流量（桶中有累积令牌时），漏桶严格控制流出速率（匀速处理）。Guava RateLimiter 和 Sentinel 都基于令牌桶实现。

**深度分析：**

```
漏桶算法（Leaky Bucket）：
    请求 → [┌─────────┐] → 匀速流出
    请求 → [│  桶/队列  │] → 固定速率处理
    请求 → [└─────────┘]
           容量固定，溢出丢弃
    特点：流出速率恒定，无法应对突发流量

令牌桶算法（Token Bucket）：
    ┌─────────────┐
    │ 固定速率放入令牌 │ → 每 r 秒放入 1 个令牌
    └─────────────┘
           ↓
    ┌─────────────┐
    │   令牌桶     │ ← 容量上限 b
    │ [🪙][🪙][🪙] │
    └─────────────┘
           ↓ 有令牌就放行
    请求 → 取走 1 个令牌 → 放行
    请求 → 无令牌 → 丢弃/排队
    特点：允许突发流量（桶中累积的令牌可以一次用完）
```

**代码示例 — Guava RateLimiter（令牌桶）：**

```java
// 平滑突发限流（SmoothBursty）
RateLimiter limiter = RateLimiter.create(10);  // 每秒 10 个令牌

public void handleRequest() {
    // tryAcquire 非阻塞，获取不到立即返回 false
    if (limiter.tryAcquire(500, TimeUnit.MILLISECONDS)) {
        // 获取到令牌，处理请求
        processRequest();
    } else {
        // 限流处理：返回提示/降级
        throw new RateLimitException("系统繁忙，请稍后重试");
    }
}

// acquire 会阻塞等待直到获取到令牌
limiter.acquire();  // 阻塞式
```

**限流维度与方案对比：**

| 维度 | 方案 | 适用场景 |
|------|------|----------|
| 单机限流 | Guava RateLimiter | 单个服务节点 |
| 单机限流 | Semaphore | 并发数控制 |
| 分布式限流 | Redis + Lua（滑动窗口） | 多节点统一限流 |
| 分布式限流 | Sentinel（阿里巴巴） | 微服务全局限流 |
| 网关层限流 | Nginx limit_req | 接入层限流 |
| 网关层限流 | Spring Cloud Gateway | 微服务网关 |

**关键细节：**

| 特性 | 漏桶 | 令牌桶 |
|------|------|--------|
| 流出速率 | 恒定 | 可变（允许突发） |
| 突发流量 | 不支持（直接丢弃/排队） | 支持（消耗累积令牌） |
| 实现复杂度 | 简单 | 稍复杂 |
| 典型应用 | 流量整形（网络设备） | API 限流（Guava、Sentinel） |
| 空闲时 | 无累积 | 令牌累积（可应对突发） |

:::danger 面试追问
- 固定窗口和滑动窗口计数器有什么问题？→ 固定窗口存在临界点突刺问题（窗口边界瞬间 2 倍流量）；滑动窗口更平滑但实现稍复杂
- Sentinel 有哪些限流规则？→ 流量控制（QPS）、热点参数限流、熔断降级、系统保护（CPU/Load 自适应）
- 分布式限流怎么做？→ Redis + Lua 原子操作（INCR + EXPIRE），或 Sentinel 集群流控模式
:::

## ⭐ 微服务如何合理拆分？拆分的粒度和原则是什么？

**简要回答：** 微服务拆分不是越细越好，核心原则是"高内聚、低耦合"，按业务领域（DDD 领域驱动设计）拆分，每个服务对应一个独立的业务能力。拆分粒度以"一个团队（Two-Pizza Team，6-10 人）能独立开发和维护"为参考标准。

**深度分析：**

```
微服务拆分原则：

1. 单一职责原则（SRP）
   每个服务只负责一个业务领域
   ✅ 订单服务、用户服务、支付服务
   ❌ 电商服务（包揽一切）

2. 业务领域驱动（DDD）
   按限界上下文（Bounded Context）划分
   ┌─────────────┐
   │   电商平台    │
   ├──────┬──────┤
   │用户域 │ 商品域 │ 订单域 │ 支付域 │ 营销域 │
   └──────┴──────┘
   每个域内部有独立的聚合根、实体、值对象

3. 数据隔离原则
   每个服务有独立的数据库
   ✅ 用户库、订单库、商品库（独立）
   ❌ 所有服务共享一个大库

4. 演进式拆分（不要一步到位）
   单体 → 模块化 → 拆分核心服务 → 逐步细化
```

**拆分粒度的判断标准：**

| 指标 | 过粗 | 过细 | 合理 |
|------|------|------|------|
| 团队规模 | 多个团队维护一个服务 | 一个人维护多个服务 | 一个服务一个团队 |
| 代码规模 | 10 万行以上 | 几百行 | 几千到几万行 |
| 修改频率 | 每次发布影响全局 | 改个小功能要改 5 个服务 | 独立变更、独立部署 |
| 服务间通信 | 几乎没有 | 大量 RPC 调用 | 少量必要的 RPC |
| 数据库 | 共享数据库 | 每个表一个服务 | 按领域独立数据库 |

**常见拆分反模式：**

```
❌ 分布式单体（Distributed Monolith）
   - 服务拆了但数据库还是共享的
   - 服务间同步调用链路过长（A → B → C → D → E）
   - 修改一个功能要同时改多个服务并协调发布

❌ 过度拆分（Nanoservices）
   - 几十上百个微服务，运维成本爆炸
   - 服务间通信开销 > 业务处理开销
   - 团队频繁跨服务联调

✅ 正确做法
   - 从单体开始，渐进式拆分
   - 优先拆分变化最频繁的模块
   - 拆分后服务间通过事件驱动解耦（MQ）
   - API Gateway 统一入口，对前端屏蔽拆分细节
```

:::danger 面试追问
- 服务拆分后分布式事务怎么处理？→ Saga 模式（补偿事务）、TCC（Try-Confirm-Cancel）、本地消息表（最终一致性）
- 如何处理服务间的数据依赖？→ 通过 API/事件进行数据同步，避免跨库 JOIN；CQRS 模式构建查询视图
- 微服务拆分后怎么做数据聚合查询？→ API 组合模式、CQRS + 读库、数据同步到 ES
:::
